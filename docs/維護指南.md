# 網站內容維護指南

這份文件說明：哪些網站內容**可以自己用瀏覽器在 GitHub 上修改、不需要寫程式**，以及一步步怎麼做。

適用對象：行銷／PM 等非工程夥伴。看不懂的地方隨時找工程師，不要硬猜。

## 先讀：三個最重要的原則

1. **只在這個 repo 改內容**：`kiwissec.github.io`（對外上線的網站）。內容（課程、服務、消息、見證）的「正本」就在這裡，你的修改不會被覆蓋。
2. **一律「開 PR」，不要直接存到 main**：在 GitHub 存檔時會問你要不要直接提交，請**務必選擇開新分支與 Pull Request（PR）**。PR 會自動跑檢查、由工程師把關，避免把壞掉的內容直接推上線。
3. **等檢查綠勾、請工程師合併**：PR 開好後，下方會自動跑檢查。出現綠勾代表內容格式正確；出現紅叉代表有錯（多半是標點），點進去看訊息或找工程師。合併後約 2–3 分鐘自動上線。

## 可以維護的四個檔案

全部在 `src/data/` 資料夾，都是 `.json` 檔（純文字、用標點分隔的資料）：

| 檔案 | 對應網站內容 | 在哪一頁 |
| --- | --- | --- |
| `src/data/courses.json` | 企業資安課程 | 課程頁、首頁課程區 |
| `src/data/services.json` | 服務項目 | 服務頁、首頁服務區 |
| `src/data/news.json` | 最新消息 | 最新消息頁 |
| `src/data/testimonials.json` | 學員見證 | 首頁見證區 |

### courses.json（課程）

每一門課是一個 `{ ... }` 區塊，欄位如下：

| 欄位 | 說明 | 範例 |
| --- | --- | --- |
| `id` | 課程的唯一代號，英數與連字號；**不可與其他課重複** | `"aw-14"` |
| `order` | 顯示順序，數字小的排前面；**每門課要不同** | `26` |
| `cat` | 分類，**只能填這三個其中一個** | `"awareness"` / `"offensive"` / `"defensive"` |
| `title` | 課程名稱 | `"資安意識培訓"` |
| `tags` | 標籤清單，中括號內用逗號分隔；**至少一個** | `["資安意識","防詐"]` |
| `desc` | 課程簡介 | `"從實際案例帶你…"` |
| `hours` | 時數（文字即可） | `"1~3 小時"` |

`cat` 三個分類的中文對照：`awareness` = 資安意識、`offensive` = 攻防演練、`defensive` = 資安防護。

### services.json（服務）

| 欄位 | 說明 |
| --- | --- |
| `id` | 唯一代號（如 `"education"`），不可重複 |
| `order` | 顯示順序，數字小的排前面，每筆不同 |
| `icon` | 圖示代碼（Font Awesome），一般不用改，要換請找工程師 |
| `name` | 服務名稱 |
| `short` | 一句話短說明 |
| `desc` | 完整說明 |
| `points` | 重點清單（中括號內用逗號分隔），至少一個 |
| `cta` | 按鈕清單；改按鈕**文字**（`label`）可以，改**連結**（`href`）或結構建議找工程師 |

### news.json（最新消息）

| 欄位 | 說明 | 範例 |
| --- | --- | --- |
| `id` | 唯一代號，建議用該則日期 | `"2026-01-31"` |
| `tag` | 標籤文字（會變成可篩選的分類） | `"新聞發佈"` |
| `date` | 日期，**格式必須是 `年-月-日`**；網站依日期由新到舊排序 | `"2026-01-31"` |
| `img` | 圖片**檔名**（只填檔名，不要路徑）；圖片要先上傳（見下方特例） | `"260131.webp"` |
| `title` | 標題 | |
| `desc` | 內容說明 | |

### testimonials.json（學員見證）

| 欄位 | 說明 |
| --- | --- |
| `id` | 唯一代號（如 `"t7"`），不可重複 |
| `order` | 顯示順序，數字小的排前面，每筆不同 |
| `q` | 見證內容（學員的話） |
| `who` | 身分（如 `"軟體工程師"`） |
| `icon` | 圖示代碼，一般不用改 |

## 怎麼修改（GitHub 網頁步驟）

### A. 修改現有文字（最常見）

1. 打開 `kiwissec.github.io` repo，進入 `src/data` 資料夾，點要改的檔（例如 `news.json`）。
2. 點右上角的鉛筆圖示（Edit this file）。
3. 直接改文字。**保持原本的引號、逗號、括號不要動到。**
4. 右上角點 `Commit changes...`。
5. 在跳出的視窗選 **「Create a new branch for this commit and start a pull request」**（不要選 Commit directly to the main branch）。
6. 點 `Propose changes` → `Create pull request`。
7. 等 PR 下方的檢查跑完：綠勾＝沒問題；紅叉＝點進去看哪裡錯。
8. 請工程師 review 並 Merge。合併後約 2–3 分鐘自動上線。

### B. 新增一筆

以新增一門課為例：

1. 編輯對應檔（如 `courses.json`）。
2. 複製一個現有的 `{ ... }` 整段，貼在清單裡。
3. 改掉裡面的 `id`、`order`、`title`、`desc` 等欄位。
4. **檢查標點**：每一筆 `{ ... }` 之間要用逗號 `,` 隔開；但**整份清單最後一筆後面不要加逗號**。
5. 後續存檔、開 PR 步驟同上。

### C. 刪除一筆

把該筆 `{ ... }` 整段刪掉，並確認剩下的逗號正確（筆與筆之間有逗號、最後一筆後面沒有逗號）。

### D. 調整顯示順序

- 課程／服務／見證：改 `order` 數字（小的排前面），維持每筆不同。
- 最新消息：改 `date`（網站自動依日期由新到舊排序）。

## 特例：新增「最新消息」要先上傳圖片

`news.json` 的 `img` 只填檔名，對應的圖片必須存在於 `public/assets/news/`：

1. 進入 `public/assets/news/` 資料夾 → 點 `Add file` → `Upload files`。
2. 拖入圖片（建議 `.webp` 格式），同樣用開 PR 的方式提交。
3. 回到 `news.json`，把 `img` 填成剛上傳的檔名（例如 `"260131.webp"`）。

可以把「上傳圖片」和「改 news.json」放在同一個 PR 一起送出。

## 不要碰的東西（需要找工程師）

- `src/data/site.ts`、`src/data/course-cats.ts`（網站固定文字與分類設定）。
- `src/components/`、`src/layouts/`、`src/styles/` 與各種設定檔。
- 不要更動既有資料的 `id` 值；`cat` 只能用 `awareness` / `offensive` / `defensive` 三個值。

## 常見錯誤與排查

- **標點要用半形**：文字要用半形雙引號 `"` 包住（不是中文的 `「」` 或 `“”`）；項目之間用半形逗號 `,`。
- **多餘或缺少的逗號**：清單最後一筆後面不要逗號；筆與筆之間不能漏逗號。這是最常見的紅叉原因。
- **`cat` 填錯**：只能是 `awareness` / `offensive` / `defensive`。
- **`date` 格式錯**：必須是 `年-月-日`（例如 `2026-01-31`）。
- **新增消息忘了上傳圖片**，或 `img` 填了路徑（只能填檔名）。
- **看紅叉訊息**：PR 頁面下方 `Checks` → 點 `Details`，會指出是哪一筆、哪個欄位有問題。

## 需要協助

把 PR 的連結貼給工程師即可。不確定能不能改的內容，先問再改，避免上線出錯。
