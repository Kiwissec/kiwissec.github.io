# 網站內容維護指南

這份文件說明如何用瀏覽器上的後台介面（CMS）維護網站內容——**不需要寫程式、也不用直接改 JSON**。

適用對象：行銷／PM 等非工程夥伴。看不懂的地方隨時找工程師，不要硬猜。

> **前置設定（工程師）**：CMS 第一次啟用前，工程師需先完成本文末〈前置設定（工程師專用）〉的 OAuth 與分支設定，夥伴才能登入使用。在那之前，內容仍可由工程師依〈直接改檔（工程師備援）〉維護。

## 先讀：三個最重要的原則

1. **後台不會直接動到正式網站**：你的每次修改都會自動送進一個「審查用的 Pull Request（PR）」，由工程師確認後才上線——所以放心編輯。
2. **等檢查綠勾**：送出後，系統會自動跑格式與內容檢查。綠勾＝沒問題；紅叉＝有錯（多半是必填漏填），點進去看訊息或找工程師。
3. **由工程師合併上線**：綠勾後請通知工程師按 Merge，約 2–3 分鐘自動上線。

## 怎麼進入後台

1. 開啟 **<https://kiwissec.github.io/admin/>**。
2. 點 **Sign in with GitHub**，用「已被授權」的 GitHub 帳號登入（沒有權限請先找工程師加入）。
3. 登入後左側會看到四個內容區：**課程、服務、最新消息、學員見證**。

## 怎麼編輯（後台操作）

### A. 修改現有內容（最常見）

1. 點左側的內容區（例如「最新消息」）。
2. 點要修改的那一筆，進入表單。
3. 直接在欄位裡改文字。下拉、日期等欄位用點選的，不用記格式。
4. 右上角點 **Save**。系統會把這次修改送進審查 PR。
5. 通知工程師檢視並合併。

### B. 新增一筆

1. 進入內容區，點右上角 **New（新增）**。
2. 逐欄填寫（必填欄位會標示）。**`代號 (id)` 必須唯一**，會成為這筆資料的檔名。
3. **Save** → 送進審查 PR。

### C. 刪除一筆

1. 進入該筆資料，於右上角選單點 **Delete**。
2. **Save / 確認** → 送進審查 PR。

### D. 調整顯示順序

- 課程／服務／見證：改 **顯示順序**（數字小的排前面），維持每筆不同。
- 最新消息：改 **日期**（網站自動依日期由新到舊排序）。

## 各內容區欄位說明

下列欄位名稱與後台表單上的標示一致；後台每個欄位下方也有提示文字。

### 課程（courses）

| 欄位 | 說明 |
| --- | --- |
| 代號 (id) | 唯一代號，英數與連字號（例如 `aw-14`）；不可重複，會成為檔名 |
| 顯示順序 | 數字小的排前面，每門課不同 |
| 分類 | **下拉選一個**：資安意識／攻防演練／資安防護 |
| 課程名稱 | 課程標題 |
| 標籤 | 逐項輸入，至少一個 |
| 課程簡介 | 一段說明文字 |
| 時數 | 文字即可（例如 `1~3 小時`） |

### 服務（services）

| 欄位 | 說明 |
| --- | --- |
| 代號 (id) | 唯一代號（例如 `education`），不可重複 |
| 顯示順序 | 數字小的排前面，每筆不同 |
| 圖示 (Font Awesome) | 圖示代碼，一般不用改；要換請找工程師 |
| 服務名稱 / 一句話短說明 / 完整說明 | 文字 |
| 重點清單 | 逐項輸入，至少一個 |
| 按鈕 | 可有多顆；每顆填**按鈕文字**與**連結**，並可勾選「主要按鈕」「外部連結」 |

### 最新消息（news）＋ 上傳圖片

| 欄位 | 說明 |
| --- | --- |
| 代號 (id) | 建議用該則日期（例如 `2026-01-31`），不可重複 |
| 標籤 | 文字；會變成最新消息頁可篩選的分類 |
| 日期 | 用日期選擇器；網站依日期由新到舊排序 |
| 圖片檔名 | **只填檔名**（例如 `260131.webp`），不要填路徑 |
| 標題 / 內容說明 | 文字 |

**上傳圖片步驟**：先用後台上方的 **媒體庫（Asset Library）** 把圖片上傳到 `assets/news`，再回到「圖片檔名」欄位填入剛上傳的檔名。建議使用 `.webp` 格式。（若在後台找不到媒體庫，請工程師協助，或改用 GitHub 上傳到 `public/assets/news/`。）

### 學員見證（testimonials）

| 欄位 | 說明 |
| --- | --- |
| 代號 (id) | 唯一代號（例如 `t7`），不可重複 |
| 顯示順序 | 數字小的排前面，每筆不同 |
| 見證內容 | 學員的話 |
| 身分 | 例如 `軟體工程師` |
| 圖示 (Font Awesome) | 一般不用改 |

## 送出之後會發生什麼（審查與上線）

1. 你按 Save → 後台把修改 commit 到 `content` 分支，並自動開一個 `content` → `main` 的 PR。
2. PR 會自動跑檢查（型別、內容稽核、頁面測試）。**綠勾＝可上線**。
3. 工程師檢視 PR、確認綠勾後 **Merge**。
4. 合併到 `main` 後自動建置部署，約 2–3 分鐘線上可見。

## 常見問題

- **送出後出現紅叉**：點 PR 頁面下方 `Details` 看哪個欄位有問題（多半是必填漏填或 `代號 (id)` 重複），修正後再 Save。
- **看不到某個內容區或無法登入**：多半是權限尚未開通，找工程師。
- **改了沒上線**：確認 PR 已被工程師 Merge；合併後才會部署。

## 直接改檔（工程師備援）

內容以「每筆一個 JSON 檔」存放於 public repo 的 `src/data/<內容區>/<id>.json`（例如 `src/data/news/2026-01-31.json`）。工程師可直接在 GitHub 編輯這些檔（請走 PR，不要直接改 `main`）。新增圖片放 `public/assets/news/`。Zod schema 與內容稽核（`npm run audit:content`）會在 PR 檢查時把關。

## 前置設定（工程師專用）— OAuth 與分支

CMS 是純前端，直接呼叫 GitHub API；唯一需要的伺服器件是一個無狀態 OAuth Worker。一次性設定如下：

1. **建立 `content` 分支並保護 `main`**（public repo `kiwissec.github.io`）
   - 從 `main` 建立長期分支 `content`。
   - 對 `main` 設 branch protection：要求透過 PR 合併、要求狀態檢查（CI verify）通過。

2. **建立 GitHub OAuth App**（GitHub → Settings → Developer settings → OAuth Apps → New）
   - Homepage URL：`https://kiwissec.github.io`
   - Authorization callback URL：`https://<你的-worker>.workers.dev/callback`
   - 取得 **Client ID**，並產生 **Client Secret**（勿寫進 repo）。

3. **部署 OAuth Worker**（Cloudflare Workers，`sveltia/sveltia-cms-auth`）
   - 參考官方說明：<https://github.com/sveltia/sveltia-cms-auth>
   - 設定 Worker 環境變數：`GITHUB_CLIENT_ID`、`GITHUB_CLIENT_SECRET`、`ALLOWED_DOMAINS`（填 `kiwissec.github.io`）。
   - 取得 Worker 網址。

4. **把 Worker 網址填進 `public/admin/config.yml`**
   - 將 `backend.base_url` 由預設的 placeholder 改為你的 Worker 網址（不含 `/callback`）。

5. **授予編輯者權限**
   - 把夥伴加為 public repo 的協作者（write），或透過 organization team 授權。

**本機驗證（OAuth 前，可選）**：Sveltia 不需 proxy server（官方不支援 `netlify-cms-proxy-server`／`decap-server`，`local_backend` 會被忽略）。要在設定 OAuth 前先驗證表單，請用 **Chromium 系瀏覽器**（Chrome／Edge／Brave）執行 `npm run dev`，開 `http://localhost:4321/admin/`，點 **Work with Local Repository** 選 repo 根目錄，即可直接讀寫本機檔案測試四個內容區（Sveltia 的 File System Access API 工作流；Firefox／Safari 不支援）。

完成後，夥伴即可於 <https://kiwissec.github.io/admin/> 登入。**首次設定後請工程師親自登入後台、對四個內容區各做一次新增／編輯／刪除測試**，確認 widget 與送出流程正常（本專案導入 PR 未對線上 OAuth 流程做實機驗證，僅驗證了資料結構、build、稽核與測試）。

## 需要協助

把 PR 的連結貼給工程師即可。不確定能不能改的內容，先問再改，避免上線出錯。
